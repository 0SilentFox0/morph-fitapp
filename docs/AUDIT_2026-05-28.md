# Аудит стану проекту FitApp

_Дата: 2026-05-28_

## Контекст

Запит: оцінити поточний стан кодової бази FitApp (React Native + Expo 55, React 19, Zustand, react-hook-form + zod), знайти слабкі місця, дублі коду та компоненти-кандидати на шарінг. Дані зібрано трьома паралельними Explore-агентами та перевірено вибірковими grep/wc-командами.

Це **аудит з пріоритетами**, не план виконання. Кожна знахідка має шлях файлу, опис, оцінку зусиль (S = години, M = день, L = декілька днів) та пріоритет (High/Medium/Low).

---

## Загальна оцінка

**Сильні сторони:**
- Чиста архітектура: чітке розділення `screens` / `components` / `navigation` / `store` / `services` / `theme`
- Централізовані дизайн-токени в `src/theme/` (colors, spacing, typography) — фундамент уже є
- Навігація типізована (`src/navigation/types.ts`)
- Zustand stores сфокусовані за доменами (app, chat, programs, sessions, draftProgram, exercise)
- Forms на react-hook-form + zod — гарний вибір
- Існує базова UI-бібліотека (17 компонентів в `src/components/ui/`)

**Загальні слабкі місця:**
- 5 файлів >440 рядків (3 з них — форми/екрани з 542-698 рядків)
- 121 хардкоднутий `borderRadius:` у StyleSheet — токени для радіусів відсутні
- Дублі форматерів (`formatDate`, `formatTime`), валідаторів, constants (TRAINING_TYPES vs TAG_OPTIONS)
- Чотири дублювання pattern-у "вибір однієї опції" в onboarding без shared ChoiceCard
- ~14% коду в TypeScript-файлах припадає на 5 найбільших файлів — концентрація складності

## 5. Performance & re-renders (фокус High)

### 5.1 Множинні zustand-селектори в одному компоненті — **High, S**
`src/screens/home/screens/AddToLibraryFormScreen.tsx:156-163` — **5+ окремих** `useDraftProgramStore((s) => s.X)` для title, tag, description, exercises та сетерів. Кожен селектор створює окрему підписку; зміна будь-якого поля → rerender всього компонента.

**Фікс:** об'єднати в один селектор з shallow-компаратором (`zustand/shallow`):
```ts
import { useShallow } from 'zustand/react/shallow';
const { title, tag, description, setTitle, setTag, setDescription } =
  useDraftProgramStore(useShallow((s) => ({ title: s.title, tag: s.tag, ... })));
```

### 5.2 Wholesale store subscription — **High, S**
- `src/screens/onboarding/steps/ChooseRoleScreen.tsx:52` — `const { setUserRole } = useAppStore()` (підписка на весь store)
- `src/screens/onboarding/steps/YoureAllSetScreen.tsx:13` — `const { setOnboarded, addPoints } = useAppStore()`
- `src/screens/home/screens/ProfileScreen.tsx:17-28` — деструктуризація 10 полів з `useOnboardingStore()` без селектора (новий обʼєкт щоразу)

При зміні будь-якого поля (points, userName) компонент рендериться. Замінити на per-field селектори або `useShallow`.

### 5.3 `ExerciseCard` без `React.memo` всередині `.map()` — **High, S**
`src/screens/home/screens/AddToLibraryFormScreen.tsx:280` — `exercises.map((ex) => <ExerciseCard key={ex.id} exercise={ex} />)`, а компонент визначений на рядках 39-145 без `React.memo`. Зміна одного set-у → rerender усіх карток.

**Фікс:** експортувати з `React.memo`. Це також гарний привід винести у `src/components/ui/ExerciseCard.tsx` (див. п. 2.2).

### 5.4 FlatList без оптимізаційних props — **High, S**
`src/screens/home/screens/GalleryScreen.tsx:266-283` — 2-колонна сітка вправ з API, але немає `initialNumToRender`, `windowSize`, `removeClippedSubviews`. При 100+ елементах — пам'ять і scroll-lag.

**Фікс:** додати `initialNumToRender={12} windowSize={5} removeClippedSubviews maxToRenderPerBatch={10}`.

### 5.5 Експресивна фільтрація без useMemo — **Medium, S**
`src/screens/home/screens/GalleryScreen.tsx:124` — `const displayExercises = filteredExercises()` викликається при кожному рендері (фільтрація + lowercase + includes по всіх вправах). Обернути в `useMemo([searchQuery, selectedCategory, exercises])`.

Подібно: `filteredExercises` визначений як **метод store**, а не selector — це антипатерн зустрічі. Краще винести у компонент через `useMemo` або зробити computed-selector з shallow.

### 5.6 Inline-callback'и блокують мемоізацію дочірніх — **Medium, S**
- `SessionFormScreen.tsx:277` — `onChangeText={(t) => { setClientSearch(t); setShowClientResults(true); }}`
- `SessionFormScreen.tsx:386-415` — `.map()` з inline `onPress`
- `HomeScreen.tsx:216-223` — `onPress={() => navigation.navigate(...)}` для ScheduleCard

Обгорнути в `useCallback`. Це матиме ефект тільки після того, як дочірні компоненти будуть `React.memo`.

### 5.7 Inline-обʼєкти у style props — **Low, S**
- `SessionFormScreen.tsx:179` — `style={[styles.header, { paddingTop: ... }]}` — новий обʼєкт щоразу
- `SessionFormScreen.tsx:263` — `hitSlop={{ top: 8, ... }}` — новий обʼєкт щоразу

Винести в `useMemo` або константу за межі компонента. Невелика економія, але easy win.

### 5.8 `react-native-chart-kit` ререндерить chart-и — **Medium, M**
`src/screens/stats/BusinessAnalyticsScreen.tsx:118-142` — `Dimensions.get('window').width` + `mockAnalyticsData.incomeOverTime` передаються в `LineChart` без мемоізації. При кожному перемиканні tab — повний reconciliation chart (200-500ms). Обгорнути дані та width в `useMemo`.

### 5.9 Що **не** виявилось проблемою

## 6. TypeScript & error handling (фокус High)

### 6.1 `tsconfig.json` — мінімальна конфігурація — **Medium, S**
Поточний файл (`/Users/silentfox/Work/fitapp/tsconfig.json`):
```json
{ "extends": "expo/tsconfig.base", "compilerOptions": { "strict": true } }
```
`strict: true` вже вмикає 7 strict-прапорів. Бракує:
- `noUncheckedIndexedAccess` — `array[i]` стане `T | undefined`, ловить багато null-помилок
- `exactOptionalPropertyTypes` — розрізнення `prop?: T` vs `prop: T | undefined`
- `noImplicitReturns`, `noFallthroughCasesInSwitch`
- `noUnusedLocals`, `noUnusedParameters` — для cleanup

### 6.2 `catch (e: any)` маскує помилки — **High, S**
`src/store/exerciseStore.ts:47` — `catch (e: any) { set({ error: e.message, ... }) }`. Якщо помилка не Error — `e.message` буде undefined, error-state стане undefined без явного дебагу.

**Фікс:**
```ts
catch (e) {
  const msg = e instanceof Error ? e.message : String(e);
  set({ error: msg, loading: false });
}
```

### 6.3 Swallowed errors у store — **High, S**
`src/store/exerciseStore.ts`:
- Рядок 64 — `loadMore`: `catch {} set({ loadingMore: false })` (без логування, без error-state)
- Рядок 73-75 — `loadCategories`: `catch { /* keep empty */ }` (помилка завантаження категорій повністю ігнорується)

`src/services/exerciseApi.ts:102` — `searchExercises` повертає `[]` при `!res.ok` без сигналу про помилку.

**Фікс:** додати щонайменше `console.error()` + опціональне `errorState`. Якщо силою сценарій (fail-silently) — задокументувати чому.

### 6.4 Fetch без timeout — **High, S**
`src/services/exerciseApi.ts` — три fetch-виклики без `AbortController`/timeout. Запит до wger.de може зависнути → loading назавжди true.

**Фікс:**
```ts
const controller = new AbortController();
const t = setTimeout(() => controller.abort(), 10000);
try {
  const res = await fetch(url, { signal: controller.signal, headers: ... });
  ...
} finally { clearTimeout(t); }
```
Винести в shared `apiClient.ts` коли з'явиться 2-й API.

### 6.5 Type assertion без runtime-перевірки — **Medium, S**
`src/services/exerciseApi.ts:74` — `(data.results as ApiExerciseInfo[]).map(mapExercise)` — `data.results` приведено як `any`-shape без перевірки. Якщо API раптом верне `null` або інший формат — runtime crash.

Аналогічно рядок 92 — `data.results as ExerciseCategory[]`.

**Фікс:** додати zod-схеми (бібліотека `zod` вже у `dependencies`):
```ts
const ApiResponse = z.object({ results: z.array(ApiExerciseInfoSchema), count: z.number(), next: z.string().nullable() });
const parsed = ApiResponse.parse(data);
```

### 6.6 Подвійний cast — **Medium, S**
`src/screens/home/screens/AddToLibraryFormScreen.tsx:565` — `width: '45%' as unknown as number` — небезпечно: маскує тип RN style. Виправити на `flex: 1` або переробити layout.

### 6.7 Error boundaries — повністю відсутні — **High, M**
`grep -rn "componentDidCatch\|ErrorBoundary" src` — 0 співпадінь. Будь-яка непіймана помилка в дереві крешить app.

**Фікс:** додати `src/components/ErrorBoundary.tsx` (class component з `componentDidCatch`) та обгорнути головний `RootNavigator`. У production надсилати в Sentry/інший logger.

### 6.8 Loading/Error/Empty states покриття — **Medium, M**

| Екран | Loading | Error | Empty |
|---|---|---|---|
| GalleryScreen | ✅ | ✅ + retry | ✅ |
| TrainingLibraryScreen | ❌ | ❌ | ✅ |
| ChatListScreen | ❌ | ❌ | ✅ |
| ClientsListScreen | ❌ | ❌ | (не перевірено) |

Зараз mock-екрани не потребують loading, але як тільки переходитимуть на API — буде проблема. Передбачити EmptyState/ErrorState компоненти зараз (див. п. 1.2).

### 6.9 Установлені, але невикористані залежності — **Medium, S**
`grep -rn "react-hook-form\|useForm\|Controller" src` — **0 співпадінь**.

- `@hookform/resolvers` (5.2.2) — те саме
- `zod` (4.3.6) — встановлено, не використовується (хоча корисно для пункту 6.5)

Усі форми (SessionFormScreen, AddToLibraryFormScreen, CardioClassFormScreen, профілі) ведуть форму через ручні `useState` + локальна валідація типу `if (title.trim().length < 2)`.

**Дві опції:**
1. Видалити невикористані залежності, документувати ручний form-pattern
2. Реально інтегрувати react-hook-form + zod (рекомендовано — економить код, дає schema-validation)

Це найвідчутніший «розбіжний намір vs реалізація» в проекті.

### 6.10 Navigation types — переважно ок, але — **Low, S**
`src/navigation/types.ts` — типи параметрів є. Швидка перевірка: переконатися що `navigation.navigate('ExerciseDetail')` без params має `ExerciseDetail: undefined` (інакше TS не зловить пропущені params).

### 6.11 Що **не** проблема

## 7. State management & data flow (фокус High)

### 7.1 Persistence (AsyncStorage) повністю відсутня — **Critical, M**
`grep -rn "AsyncStorage\|persist" src` → 0 співпадінь. `@react-native-async-storage/async-storage` навіть **не у `package.json`**.

Наслідки:
- `appStore.isOnboarded` скидається при перезапуску → користувач проходить onboarding **щоразу**
- Дані з `onboardingStore` (ім'я, фото, типи тренувань, досвід) втрачаються
- `programsStore` / `sessionsStore` / `chatStore` повертаються до mock-початкового стану
- `draftProgramStore` (напівготова програма) втрачається при backgrounding app

**Фікс:** додати `@react-native-async-storage/async-storage` + `zustand/middleware/persist` для критичних stores: `appStore`, `onboardingStore`, можливо `draftProgramStore`. Mock-stores (programs/sessions/chat) — поки можна не персистити, але після підключення backend — обов'язково.

```ts
import AsyncStorage from '@react-native-async-storage/async-storage';
import { persist, createJSONStorage } from 'zustand/middleware';

export const useAppStore = create(persist<AppState>(
  (set) => ({ /* ... */ }),
  { name: 'app-storage', storage: createJSONStorage(() => AsyncStorage) }
));
```

### 7.2 `draftProgramStore` lifecycle — race conditions — **High, M**

Точки взаємодії:
- Ініціалізація: `GalleryScreen.tsx:46-52` (вибір вправ → draft)
- Заповнення: `AddToLibraryFormScreen.tsx:156-173`
- Save: `addProgram(draft)` → `programsStore`

Виявлені проблеми:
- **Не очищується при back-navigation** з GalleryScreen без save. Наступне відкриття покаже stale exercises.
- **`useDraftProgramStore.getState().setX()` всередині useEffect** (`AddToLibraryFormScreen.tsx:171-172`) — обхід reactivity. У edit-mode при швидкому back/forward можна отримати неконсистентний стан.
- **Немає atomic save** — `addExercises()` + `addProgram()` йдуть як два кроки, якщо щось між ними failure — частковий стан.

**Фікс:**
- Додати `resetDraft()` у `draftProgramStore`
- Викликати у `useFocusEffect` cleanup на GalleryScreen / AddToLibraryFormScreen при не-save виході
- Зробити save в одній transactional action: `programsStore.addProgramFromDraft()`

### 7.3 `exerciseStore` — відсутній debounce пошуку — **High, S**
`src/store/exerciseStore.ts:78` — `setSearchQuery` оновлює `searchQuery`, але `filteredExercises()` фільтрує **локально** (по вже завантажених — поки що).

Якщо в майбутньому search піде на API — потрібен debounce + cancel попередніх запитів (`AbortController`). Зараз — це робоча приставка, але документувати обмеження: пошук тільки серед перших 20-40 завантажених вправ.

Також: `loadCategories` викликається з GalleryScreen mount, але немає захисту від повторного завантаження якщо `categories.length > 0`.

### 7.4 Cross-store coupling через `getState()` — **Medium, S**
`AddToLibraryFormScreen.tsx:171-172` — `useDraftProgramStore.getState().setDescription(...)` — bypass-ить React reactivity. Працює, але кращий патерн: отримати сетер через звичайний селектор і викликати в ефекті.

`GalleryScreen.tsx:110-120` — координація `draftProgramStore.addExercises()` → `programsStore.addProgram()` без транзакційності (див. 7.2).

### 7.5 Source-of-truth конфлікти — **Medium, S**
- `src/mocks/data.ts` містить `mockTrainingPrograms`
- `src/store/programsStore.ts` має `initialPrograms` (підозра на дублювання — потребує перевірки під час фіксу)
- `SessionFormScreen.tsx:24` імпортує `mockClients` напряму з mocks, обходячи store

**Фікс:** один source-of-truth — store. Mock-дані використовуються лише як initial state у store-конструкторі. Жоден екран не імпортує з `mocks/` напряму.

### 7.6 Onboarding — нема resumability — **Medium, M**
`onboardingStore.ts` зберігає поля (name, profilePhotoUri, trainingTypes, etc.), але **немає `currentStep`**. `appStore.isOnboarded` — лише binary flag.

Сценарій: користувач закрив app на 5-му кроці з 13 → відкрив → починає з кроку 1. Усі введені дані залишаться в store (якщо додамо persistence з п. 7.1), але крок завжди буде перший.

**Фікс:** додати `currentStepIndex: number` в `onboardingStore`. `OnboardingNavigator` читає його при mount і `navigation.navigate(stepName)`.

### 7.7 API config hardcoded — **Low, S**
`src/services/exerciseApi.ts:1` — `const BASE_URL = 'https://wger.de/api/v2'`. Винести у `src/config/env.ts`:
```ts
export const API_BASE = __DEV__ ? 'https://wger.de/api/v2' : process.env.EXPO_PUBLIC_API_BASE ?? 'https://wger.de/api/v2';
```
Зараз не критично (один публічний API), але після додавання власного backend стане необхідним.

### 7.8 Async loading-state корректність — **Low, S**
`exerciseStore.loadExercises` — правильно встановлює `loading: false` у try/catch. Але `loadMore` ніколи не встановлює `error` — failure тихо стопить пагінацію без сигналу UI.

## 9. Що ще не покрито в цьому аудиті

Щоб бути чесним — навіть після поглиблення є дімензії, які не торкались:

- **Accessibility** — `accessibilityLabel`, focus-управління, screen-reader-friendly
- **i18n** — всі рядки англійською інлайн; немає i18n-бібліотеки
- **Tests** — `find -name "*.test.*"` дає 0; нема Jest-конфігу взагалі
- **CI/CD** — `.github/workflows` не перевірявся
- **Security** — sensitive logging, deep-link valid-я, image-permissions у `app.json`
- **Memory leaks** — useEffect cleanup на listeners/timers не перевірявся exhaustively
- **iOS vs Android edge cases** — `Platform.OS` гілки не аудилися
- **Bundle size** — `react-native-chart-kit`, `expo-blur`, `expo-document-picker` — чи всі реально використовуються
- **Image performance** — RN `Image` без cachePolicy; для production з фото клієнтів варто `expo-image`

Якщо потрібен ще один прохід — скажіть, які з цих 9 розкопати далі.
